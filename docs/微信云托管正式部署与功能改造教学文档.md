# 微信云托管正式部署与功能改造教学文档

本文档面向第一次部署本项目的同学，说明如何把“校园二手交易平台”部署为：

微信小程序前端 -> 微信云托管 Flask 后端 -> 微信云托管 MySQL -> 千问 API -> 微信账号登录。

## 本轮正式优化补充

## 真实登录和开发测试登录并存

正式用户仍通过 `POST /api/v1/auth/wechat-login` 登录，后端只使用微信 `code2Session` 返回的 `openid` 作为绑定依据。昵称和头像必须由用户在“完善资料”页主动保存，不能把微信昵称头像当成账号绑定依据。

为了方便课程演示多账号交易，后端新增开发测试登录：

```text
POST /api/v1/auth/dev-test-login
Body: {"account":"buyer_a|buyer_b|admin"}
```

启用条件：

```text
DEV_TEST_LOGIN_ENABLED=1
```

生产环境和正式云托管发布建议保持：

```text
DEV_TEST_LOGIN_ENABLED=0
```

测试账号由 `backend/scripts/init_db.py` 初始化，适合演示：A 发布商品、B 购买、买卖双方聊天、管理员审核商品、管理员查看后台页面。

## 收藏页分类逻辑

收藏记录会保存 `favorited_price`。收藏页四个分类由后端接口返回：

- 全部：`GET /api/v1/favorites?type=all`
- 降价宝贝：`GET /api/v1/favorites?type=price_drop`，当前价格低于收藏时价格
- 有效宝贝：`GET /api/v1/favorites?type=valid`，在售、交易中或已售出
- 失效宝贝：`GET /api/v1/favorites?type=invalid`，卖家主动下架

失效商品可以通过 `POST /api/v1/favorites/cleanup-invalid` 清理。已售出商品不会自动清除，用户可以手动取消收藏。

## AI 文案优化

AI 仍使用真实千问 DashScope API，必须配置 `AI_MODE=qwen`、`DASHSCOPE_API_KEY`、`QWEN_MODEL`。提示词已调整为校园二手场景：标题短、自然、不堆词；描述口语化但保留重点，不虚构品牌、配件、保修、包邮或交易承诺。

## UI 和交互优化

- 发布商品页：分类、成色改为选填，图片区、输入区、价格库存区、AI 按钮和底部操作栏重新整理。
- 商品详情页：新增卖家头像昵称卡片，可进入卖家主页；底部收藏、联系卖家、立即购买主次关系更清晰。
- 聊天页：关联商品卡片可进入详情；头像可进入个人主页；气泡、输入框和商品卡片比例优化。
- 收藏页：新增四个分类栏、取消收藏、清理失效收藏。
- 完善资料页：去掉“微信资料/虚拟身份”分栏，保留微信头像昵称能力，同时支持相册和拍照设置头像。
- 消息列表、收货地址页：统一卡片、按钮、表单和列表间距。

## 个人主体限制

微信手机号授权和微信支付可能受个人主体小程序权限限制。当前项目保留正式设计和接口结构，但支付仍以 mock 闭环完成课程演示，不应在验收或演示时宣称个人主体已完整打通微信支付。


## 1. 项目目录

- `miniprogram/`：微信小程序前端，使用微信开发者工具打开。
- `backend/`：Flask 后端，部署到微信云托管。
- `backend/Dockerfile`：云托管容器构建文件。
- `backend/sql/init_db.sql`：正式 MySQL 表结构初始化脚本，执行前必须备份数据库。
- `backend/scripts/init_db.py`：课程演示数据初始化脚本或接口复用逻辑。
- `docs/`：部署、测试和课程说明文档。

## 2. 云托管环境变量

进入微信云托管控制台，选择服务 `flask-fnnj`，在服务设置或发布高级设置中填写环境变量。

必须填写：

```text
DB_BACKEND=mysql
MYSQL_ADDRESS=云托管 MySQL 地址
MYSQL_PORT=3306
MYSQL_USERNAME=root
MYSQL_PASSWORD=云托管推送给你的数据库密码
MYSQL_DATABASE=flask_demo
JWT_SECRET=一串足够长的随机密钥
INIT_TOKEN=一串初始化口令
FLASK_DEBUG=0
FLASK_HOST=0.0.0.0
FLASK_PORT=80
WECHAT_AUTH_MODE=wechat
WECHAT_APPID=微信小程序 AppID
WECHAT_SECRET=微信小程序 AppSecret
AI_MODE=qwen
DASHSCOPE_API_KEY=阿里云百炼 DashScope API Key
QWEN_MODEL=qwen-plus
PAYMENT_MODE=mock
COS_BUCKET=对象存储 bucket，可暂留空
COS_REGION=对象存储地域，可暂留空
```

注意：必须写 `MYSQL_DATABASE`，不能写成 `MYSQL_DATABAS`，最后的 `E` 不能少。

所有密码、JWT 密钥、千问 API Key、微信 AppSecret 都只能写在环境变量里，不要写进代码，也不要提交到仓库。

## 3. 端口和 Dockerfile

微信云托管发布时端口填写：

```text
80
```

容器必须监听：

```text
0.0.0.0:80
```

本项目入口是 `wsgi.py` 里的 `app`，所以云托管启动命令为：

```bash
gunicorn --bind 0.0.0.0:${PORT:-80} wsgi:app
```

如果发布时仍填 5000，会导致云托管转发端口和容器监听端口不一致。

## 4. MySQL 初始化

先在 DMS 或数据库控制台备份当前数据库。

然后打开 DMS 的 SQL 窗口，执行：

```sql
source backend/sql/init_db.sql;
```

如果 DMS 不支持 `source`，就打开 `backend/sql/init_db.sql`，复制全部 SQL 到 SQL 窗口执行。

课程演示数据可以通过云托管调试接口初始化：

```text
POST /api/v1/init-demo-data
Header: X-Init-Token = 你配置的 INIT_TOKEN
Body: {}
```

该接口只在你知道初始化口令时可用。

## 6. 小程序前端配置

小程序已在 `miniprogram/app.js` 中初始化云开发环境：

```js
wx.cloud.init({
  env: 'prod-d2g73fc4ha6d2e317',
  traceUser: true
})
```

所有普通 API 请求都通过 `miniprogram/utils/request.js` 调用：

```js
wx.cloud.callContainer({
  config: { env: 'prod-d2g73fc4ha6d2e317' },
  path: '/api/v1/...',
  header: { 'X-WX-SERVICE': 'flask-fnnj' }
})
```

不要再默认请求 `10.x.x.x:5000` 这类局域网 IP。

## 7. 微信登录验证

```text

WECHAT_APPID=你的 AppID
WECHAT_SECRET=你的 AppSecret
```

验证步骤：

2. 手机预览小程序。

## 13. 常见错误

- `MYSQL_ADDRESS` 写成 `localhost`：云托管容器会连自己，不会连到数据库。
- `MYSQL_DATABASE` 写成 `MYSQL_DATABAS`：数据库名读不到。
- 容器监听 `127.0.0.1`：云托管无法转发流量。
- 云托管端口填 5000：本项目正式配置要求填 80。
- 前端仍请求局域网 IP：同学手机不在同一 Wi-Fi 时无法访问。
- `AI_MODE` 仍为 mock：千问接口不会被正式调用。
- `DASHSCOPE_API_KEY` 未配置：AI 接口会返回配置缺失。
- `WECHAT_AUTH_MODE` 仍为 mock：不是真实微信登录。
- `WECHAT_APPID` 或 `WECHAT_SECRET` 缺失：微信 code 无法换 openid。
- MySQL 表没初始化：先执行 `init_db.sql` 或初始化接口。
- token 没携带：检查 `request.js` 是否自动加 `Authorization`。
- 云托管日志出现 500：先看启动日志，再看 `/api/v1/health/db`。
- 图片上传失败：检查上传接口和 COS 配置。
- 管理员初始化失败：检查 `INIT_TOKEN` 和管理员初始化流程。

## 本次 UI、图片和售后规则补充

本次小程序前端按 `ui-reference/01-home.png` 到 `ui-reference/05-mine.png` 的参考风格统一为清爽校园绿色视觉：主色 `#16B85E`，页面背景 `#F7F8F5`，卡片使用白底、大圆角和轻阴影，按钮、标签、输入框统一为圆润移动端比例。已重点改造首页、商品详情、发布商品、聊天、我的、订单、售后和收藏页面。

开发测试登录只在 `DEV_TEST_LOGIN_ENABLED=1` 且前端开发环境中显示。小程序登录页的一键测试账号只保留测试用户 A、测试用户 B、管理员。演示卖家账号仍可通过手机号密码登录使用，不作为前端一键测试入口展示。

正式微信登录仍以 `code2Session` 返回的 `openid` 作为唯一绑定依据。`wx.login` 只负责拿登录 code，不会自动得到用户昵称头像。真实用户首次登录后如果平台昵称或头像为空，会进入完善资料页，由用户主动设置头像和昵称。

图片持久化规则：商品图、用户头像、聊天图和售后凭证都必须在数据库中保存可恢复的 URL 或 file_key。本地 `uploads/` 只适合开发调试；云托管容器本地目录不保证重启后持久。正式演示建议配置 COS 或云存储，并在云托管环境变量中填写 `COS_BUCKET`、`COS_REGION` 以及对应访问授权。若暂未开通 COS，需要在答辩中说明当前代码保留正式接口和数据库记录结构，后续将文件存储后端切换到 COS。

售后规则：已完成订单按钮统一为“申请售后”。买家提交售后后，只有卖家可以看到同意、拒绝、部分退款按钮；买家自己不能处理自己的申请。管理员平台介入只保留支持买家退款、支持卖家拒绝、部分退款三种处理结果。退款成功后商品状态变为 `off_shelf`，不会自动恢复在售，需要卖家重新发布或重新上架。

真机测试重点：微信真实登录、测试用户 A/B/管理员切换、首页自动刷新、商品图和头像重启后可恢复、消息未读角标、本地时间显示、售后按钮权限、卖家处理售后、管理员仲裁结果、退款成功后商品下架。
